Begriffe & Übergangstechnologien

Erklärungsseite zu Aufgabe 5 · Dual Stack, Tunneling, NAT64, SLAAC, IPv6-Header und mehr

Sektion 1

Warum Übergangstechnologien?

IPv4 und IPv6 sind nicht direkt kompatibel. Ein IPv4-Gerät kann ein IPv6-Paket nicht verstehen und umgekehrt. Das Internet kann aber nicht über Nacht umgestellt werden – Milliarden von Geräten und Diensten müssen schrittweise migriert werden.

Deshalb brauchen wir drei Hauptstrategien, die den Übergang ermöglichen:

  • Dual Stack – Beide Protokolle gleichzeitig betreiben
  • Tunneling – IPv6-Pakete in IPv4-Paketen kapseln
  • Translation (NAT64) – Zwischen den Protokollen übersetzen
┌───────────┐ ┌─────────────────────┐ ┌───────────┐ │ IPv4-Welt │ ↔ │ Übergangstechnologie │ ↔ │ IPv6-Welt │ │ nur v4 │ │ Dual Stack / Tunnel │ │ nur v6 │ │ │ │ / NAT64 │ │ │ └───────────┘ └─────────────────────┘ └───────────┘
Sektion 2

Dual Stack

Bei Dual Stack laufen IPv4 und IPv6 gleichzeitig auf demselben Gerät bzw. im selben Netzwerk. Das Betriebssystem entscheidet pro Verbindung, welches Protokoll verwendet wird.

┌────────────────────────────────┐ │ Dual-Stack-Gerät │ │ │ │ ┌────────────┐ ┌────────────┐ │ │ │ IPv4-Stack │ │ IPv6-Stack │ │ │ │ 192.168.1.5│ │ 2001:db8::5│ │ │ └────────────┘ └────────────┘ │ └────────────────────────────────┘
Vorteile
  • Volle Kompatibilität mit beiden Welten
  • Einfaches Konzept
  • Schrittweise Migration möglich
Nachteile
  • Doppelte Konfiguration
  • Doppelte Ressourcen
  • Doppelte potenzielle Fehlerquellen

Praxisbezug

Die meisten modernen Betriebssysteme (Windows, macOS, Linux) und Internet-Provider nutzen heute Dual Stack. Dein Computer betreibt sehr wahrscheinlich bereits beide Protokolle gleichzeitig.

Sektion 3

Tunneling

Beim Tunneling werden IPv6-Pakete in IPv4-Pakete eingekapselt (eingepackt). So kann IPv6-Kommunikation über eine reine IPv4-Infrastruktur stattfinden.

Stell dir vor, du steckst einen Brief (IPv6) in einen zweiten Umschlag (IPv4), damit er über das alte Postsystem transportiert werden kann.

┌───────────────────────────────────────┐ │ IPv4-Header │ IPv6-Header │ Nutzdaten │ │ (Umschlag) │ (Brief) │ (Inhalt) │ └───────────────────────────────────────┘

Gängige Tunneling-Protokolle

  • 6to4 – Automatisches Tunneling mit dem Präfix 2002::/16. Leitet IPv6-Pakete über öffentliche IPv4-Adressen weiter.
  • Teredo – Tunneling durch NAT hindurch (nutzt UDP-Kapselung). Von Microsoft entwickelt. Funktioniert auch hinter NAT-Geräten.
  • ISATAP – Intra-Site Automatic Tunnel Addressing Protocol. Gedacht für den Einsatz innerhalb von Organisationen, um IPv6 über ein internes IPv4-Netzwerk zu transportieren.
Vorteile
  • Bestehende IPv4-Infrastruktur kann weitergenutzt werden
  • Ermöglicht IPv6-Inseln über IPv4 zu verbinden
Nachteile
  • Overhead durch zusätzliche Header
  • Erhöhte Komplexität
  • Potenzielle Sicherheitsprobleme
Sektion 4

NAT64 / DNS64

NAT64 übersetzt zwischen IPv6- und IPv4-Adressen. So können IPv6-only-Clients auf IPv4-only-Server zugreifen.

Wie funktioniert es?

1
Ein IPv6-Client möchte einen IPv4-Server erreichen (z.B. 93.184.216.34)
2
DNS64 erstellt einen synthetischen AAAA-Record: Die IPv4-Adresse wird in ein IPv6-Präfix eingebettet → 64:ff9b::93.184.216.34 (Präfix 64:ff9b::/96)
3
Das NAT64-Gateway empfängt das IPv6-Paket, extrahiert die IPv4-Adresse und übersetzt das Paket zwischen den Protokollen

Unterschied: NAT vs. NAT64

NAT (NAT44)
IPv4 privat → IPv4 öffentlich
Gleiches Protokoll, nur Adressübersetzung
≠
NAT64
IPv6 → IPv4
Protokollübergreifende Übersetzung

Praxisbezug

NAT64 wird vor allem in Mobilfunknetzen eingesetzt (z.B. T-Mobile). Viele Mobilfunkanbieter betreiben inzwischen IPv6-only-Netze und nutzen NAT64 für den Zugang zur IPv4-Welt.

Sektion 5

SLAAC – Stateless Address Autoconfiguration

SLAAC ermöglicht es einem Host, sich selbst eine IPv6-Adresse zu konfigurieren – ganz ohne DHCP-Server.

„Stateless“ bedeutet: Kein Server merkt sich, welche Adressen vergeben wurden. Der Host handelt eigenständig.

SLAAC-Prozess Schritt für Schritt

1
Host erstellt eine Link-Local-Adresse: fe80:: + Interface ID
2
Host sendet eine Router Solicitation (RS) an ff02::2 (alle Router auf dem Link)
3
Router antwortet mit einem Router Advertisement (RA), das das Netzwerk-Präfix enthält (z.B. 2001:db8:abcd:1::/64)
4
Host kombiniert: Präfix vom Router + selbst generierte Interface ID = vollständige Adresse
5
Host führt Duplicate Address Detection (DAD) durch, um sicherzustellen, dass die Adresse im Netzwerk einmalig ist
Host Router │ │ │──── RS (Router Solicitation) ─────────→│ an ff02::2 │ │ │←──── RA (Router Advertisement) ────────│ Präfix: 2001:db8::/64 │ │ │ Präfix + Interface ID = Adresse │ │ 2001:db8::a1b2:c3ff:fed4:e5f6 │ │ │ │──── DAD (Ist die Adresse frei?) ───────→│ │ │

Interface ID: Wie wird sie erzeugt?

  • EUI-64 – Abgeleitet aus der MAC-Adresse (siehe Aufgabe 04)
  • Privacy Extensions – Zufällig generiert, um die MAC-Adresse nicht preiszugeben
SLAAC (stateless)
Host konfiguriert sich selbst
Kein Server trackt Adressen
vs.
DHCPv6 (stateful)
Server weist Adressen zu
Server kennt alle vergebenen Adressen
Sektion 6

Der IPv6-Header

Der IPv6-Header ist gegenüber IPv4 vereinfacht worden und hat eine feste Größe von 40 Bytes. Weniger Felder bedeuten schnellere Verarbeitung durch Router.

Feld Bits Beschreibung
Version 4 Immer 6
Traffic Class 8 QoS-Priorität (Quality of Service)
Flow Label 20 Identifiziert zusammengehörige Paketflüsse
Payload Length 16 Größe der Nutzdaten in Bytes
Next Header 8 Nächstes Protokoll (TCP=6, UDP=17, ICMPv6=58)
Hop Limit 8 Wie TTL bei IPv4 – wird an jedem Router um 1 verringert
Source Address 128 Quelladresse
Destination Address 128 Zieladresse

Was FEHLT im Vergleich zu IPv4?

  • KEINE Header-Prüfsumme (Checksum) – Entfernt für bessere Performance. Die Integritätsprüfung übernehmen höhere Schichten (TCP/UDP).
  • KEINE Fragmentierungs-Felder im Basis-Header – Verschoben in optionale Extension Headers. Router fragmentieren nicht mehr.
  • KEIN Header-Längenfeld – Der Header ist immer 40 Bytes lang, daher unnötig.

Extension Headers

Zusätzliche, optionale Header werden über das Next Header-Feld verkettet. Jeder Extension Header verweist auf den nächsten. So bleibt der Basis-Header schlank und Router müssen nur den Basis-Header lesen.

Sektion 7

Integrität

Datenintegrität bedeutet: Die Daten kommen unverändert beim Empfänger an. Nichts wurde unterwegs verfälscht, beschädigt oder manipuliert.

IPv4
Header-Prüfsumme vorhanden
Muss an jedem Router neu berechnet werden (TTL ändert sich)
→
IPv6
KEINE Header-Prüfsumme
Integrität wird delegiert an TCP/UDP und optional IPsec

Wer stellt die Integrität sicher?

  • Transportschicht – TCP- und UDP-Prüfsummen (bei IPv6 ist die UDP-Prüfsumme Pflicht, bei IPv4 optional)
  • IPsec – Optional, aber IPv6 hat eingebaute Unterstützung. Bietet Authentifizierung und Verschlüsselung.

Warum wurde die Prüfsumme entfernt?

Performance: Router mussten bei IPv4 an jedem Hop die Prüfsumme neu berechnen (weil sich TTL ändert). Bei IPv6 entfällt dieser Aufwand – Router können Pakete schneller weiterleiten.

Sektion 8

Wichtige Multicast-Adressen

IPv6 nutzt Multicast intensiv – z.B. für Neighbor Discovery und SLAAC. Diese Adressen solltest du kennen:

ff02::1 All Nodes – Alle Geräte auf dem Link (wie ein lokaler Broadcast)
ff02::2 All Routers – Alle Router auf dem Link (hierhin geht die RS bei SLAAC)
ff02::1:ff00:0/104 Solicited-Node Multicast – Wird für DAD und Neighbor Discovery verwendet. Basiert auf den letzten 24 Bit der IPv6-Adresse.
Sektion 9

Tipps für die Klausur

Das Wichtigste auf einen Blick

  • Kenne die 3 Übergangsstrategien und wann welche eingesetzt wird:
    Dual Stack = beide Protokolle · Tunneling = Kapselung · NAT64 = Übersetzung
  • SLAAC-Prozess in der richtigen Reihenfolge: RS → RA → Präfix + Interface ID kombinieren → DAD
  • Der IPv6-Header hat KEINE Prüfsumme (Checksum) – im Gegensatz zu IPv4
  • IPv6-Header ist immer 40 Bytes groß (fest, keine variable Länge)
  • Next Header bei IPv6 entspricht dem Protocol-Feld bei IPv4
  • Hop Limit bei IPv6 entspricht TTL bei IPv4
  • NAT ≠ NAT64: NAT übersetzt IPv4↔IPv4, NAT64 übersetzt IPv6↔IPv4
  • Multicast-Adressen: ff02::1 (All Nodes), ff02::2 (All Routers)
